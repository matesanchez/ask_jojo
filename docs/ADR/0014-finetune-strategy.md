# ADR 0014 — Fine-Tune Strategy for the Nurix Corpus

**Status:** Accepted
**Date:** 2026-05-19
**Deciders:** Mateo de los Rios
**Related:** `ADR 0004` (local-first deployment), `ADR 0009` (DPAPI config), `ADR 0013` (standalone installer)

---

## Context

JoJo Bot v2.0 currently relies on the Anthropic API (claude-sonnet-4-6) as its synthesis backend. Every query the user sends triggers an API call. That creates three ongoing concerns:

1. **Per-query cost.** At moderate usage levels, the API cost is manageable, but at team-wide adoption across multiple departments the cost compounds quickly.
2. **Latency.** A round-trip to the Anthropic API takes 2–8 seconds depending on network conditions. This is acceptable for occasional deep lookups but too slow for interactive use in lab settings where fast turnaround matters.
3. **Data sovereignty.** Nurix's research data leaves the building on every query. For some departments or regulatory contexts this is acceptable; for others it is not.

The wiki corpus has grown to ~150 pages covering targets, programs, methods, platforms, and decisions. It is large enough to be worth fine-tuning on. The synthetic-data pipeline described in this ADR is the mechanism for turning that corpus into a training dataset.

The goal of this ADR is not to mandate running a training job immediately — that requires budget approval and hardware. The goal is to fix the strategy so that dataset generation can proceed in parallel and the training job can be initiated once conditions are met.

---

## Decision

Fine-tune on a synthetic dataset generated from the wiki corpus.

**Primary backend:** Anthropic Bedrock fine-tune targeting Sonnet 4.6 (or the Bedrock-available equivalent at the time the job runs). This is the default path because it reuses the same model family JoJo Bot already uses, minimizes integration work, and preserves response quality.

**Secondary (offline) backend:** HuggingFace LoRA fine-tune targeting either Llama-3.1-8B-Instruct or Qwen2.5-7B-Instruct. This path is for air-gapped deployments or situations where Bedrock access is unavailable or cost-prohibitive. The offline path is documented in `docs/ops/offline-model.md`.

### Synthetic dataset generation

Dataset examples are generated by the `packages/jojo_finetune` module using three example types:

- **paraphrase** — the model receives a passage from a wiki page and produces a one-sentence restatement.
- **fill_blank** — a key term is removed from a factual sentence; the model supplies the missing word or phrase.
- **synthesis** — the model is given concepts from two or more pages and asked to articulate the relationship.

Every example must carry a `citation` field pointing to one or more real wiki page slugs. Any example that cannot be traced to a specific wiki page is rejected by the dataset validator before the file is used for training.

### Synthetic-data risks and guardrails

**Model collapse risk.** Fine-tuning on model-generated text can compress the diversity of outputs, causing the fine-tuned model to converge on a narrower vocabulary and a more stereotyped response style than the base model. Guardrail: every example must be source-cited to a wiki page; examples generated without a citation anchor are discarded before the file is committed to `data/finetune/`.

**Hallucination amplification.** Fine-tuning on plausible-but-wrong examples locks errors into the model weights. Guardrail: an eval harness runs the fine-tuned model against the gold benchmark in `docs/qa/benchmark-questions.md`. Any model that achieves accuracy below 0.70 on that benchmark is rejected and the training run is flagged for review before deployment.

**Trigger criteria for running a real training job:**

1. Bedrock fine-tune budget has been formally approved.
2. The eval harness has established a baseline score for the current API-backed model on the benchmark.
3. The dataset has reached at least 1,000 examples and a diversity check (unigram entropy across outputs) shows no significant compression relative to the wiki source text.

None of these triggers implies a firm timeline. Dataset generation can proceed before all three are met; training cannot.

---

## Consequences

### Positive

- **Reduced per-query cost.** A fine-tuned Bedrock model billed at inference rates replaces the current full-size Sonnet calls. Cost reduction is estimated at 60–80% for equivalent query volume, though this depends on Bedrock pricing at the time of deployment.
- **Lower latency.** A locally hosted open-weights model eliminates the network round-trip entirely, enabling sub-100 ms response times on adequate hardware.
- **Data sovereignty option.** The HuggingFace LoRA path allows Nurix to process all queries on-premises with no data leaving the network boundary.

### Negative

- **Training run cost.** A Bedrock fine-tune job costs money upfront. A LoRA training run requires GPU hardware or a cloud GPU instance. Both are capital expenditures not currently budgeted.
- **Maintenance burden.** A fine-tuned model becomes stale as the corpus evolves. Re-training is needed whenever the wiki undergoes a major corpus update (estimated annually or when more than ~20% of pages are substantially revised).
- **Offline model accuracy gap.** Based on current benchmarks for 7–8B parameter LoRA models versus Claude, an offline fine-tuned model is likely to underperform the API-backed baseline by approximately 30–50% on the benchmark questions. The accuracy gap is acceptable for narrow, well-scoped queries but may be noticeable on synthesis questions that require multi-page reasoning.

---

## Revisit When

- Bedrock fine-tune pricing changes significantly (upward or downward).
- A new open-weights model family closes the accuracy gap to less than 15% on the benchmark.
- The wiki grows past 500 pages and dataset diversity becomes a meaningful constraint.
- A regulatory or compliance requirement explicitly prohibits API-based synthesis for any subset of queries.
